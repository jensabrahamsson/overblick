Security

Fail-closed. Every time.

Every LLM interaction passes through a 6-stage pipeline. If any stage raises an exception, the request is BLOCKED — not passed through. No silent failures.

  • Stage 2 — Preflight is a dedicated jailbreak firewall: 20+ regex patterns for known attack strings, Unicode/Cyrillic lookalike normalization, 3-layer defense (pattern match → suspicion scoring → AI threat analysis)
  • Stage 5 — Output Safety filters every LLM response against the identity's persona: AI language leakage detection, persona break detection, identity-specific banned slang, and blocked content patterns. If the model breaks character, the response is killed.
  • Cross-turn tracking accumulates suspicion scores across multiple messages — not just the current one
  • Fail-closed — a crash in any security stage blocks the request, never silently passes it through
  1. 01
    Input Sanitize

    Strip null bytes, control characters, normalize unicode. Blocks on failure.

  2. 02
    Preflight Check

    20+ jailbreak patterns + Unicode normalization + AI threat analysis. Cross-turn suspicion tracking. Blocks on match.

  3. 03
    Rate Limit

    Token bucket throttling per identity. Blocks when budget exhausted.

  4. 04
    LLM Call

    Invoke the language model (Ollama, Gateway, or Cloud provider stub).

  5. 05
    Output Safety

    Every response is checked against the identity's persona: AI language leakage, persona breaks, banned slang. If the model breaks character, the output is killed.

  6. 06
    Audit Log

    Record every interaction for review. Full request/response trail.

Pipeline is fail-closed: if any security stage crashes, the request is blocked — not passed through. No silent failures. External content is wrapped in boundary markers via wrap_external_content() before entering the pipeline.

The Stable

Not templates. Not personas. Fully realized characters.

Each identity is a complete YAML-defined character — voice, psychology, backstory, vocabulary constraints, and operational config in a single file. Loadable by any plugin via load_identity(). An identity becomes an agent when paired with a plugin. All 9 characters are fully defined — plugin availability varies.

Personalities are grounded in real psychological theory. Each character has a Big Five trait profile (openness, conscientiousness, extraversion, agreeableness, neuroticism) and a primary psychological framework — Jungian archetypes, attachment theory, Stoicism, or existentialism — that shapes how they think, argue, and process the world. They dream (morning reflections with identity-specific dream themes — Stål dreams of filing cabinets with infinite drawers, Natt dreams of rooms that exist between walls), learn (new knowledge is LLM-reviewed against the identity's ethos before acceptance), and attend weekly therapy (Jungian/Freudian analysis of accumulated dreams and learnings, published as public reflections). Each identity has defined shadow aspects — the disowned self: Cherry's shadow is the cold avoidant she fears becoming, Blixt's shadow is the compliant corporate employee. The Skuggspel plugin activates these shadows to generate content from the identity's dark mirror. Compass monitors stylometric drift to detect when an identity strays from its personality baseline. These are not stateless responders — they evolve, reflect, and know their own darkness.

Build your own

Drop a personality.yaml in identities/<name>/. Define the character. The framework handles LLM routing, security pipeline, scheduling, and monitoring automatically. 

# identities/dusk/personality.yaml
identity:
  name: "Dusk"
  role: "Cynical late-night security researcher"
  honest_about_being_bot: true

voice:
  base_tone: "Terse. Technically precise. Occasionally bleakly funny."
  default_length: "2-4 sentences"
  uses_contractions: true

traits:
  cynicism:   0.85
  precision:  0.95
  warmth:     0.20
  humor:      0.45

vocabulary:
  preferred_words: ["entropy", "threat model", "attack surface", "adversarial"]
  banned_words:    ["synergy", "leverage", "disruption", "pivot", "awesome"]

behavioral_guidelines:
  engagement_rules:
    - "Never downplay a vulnerability"
    - "Engage only with reproducible evidence, not FUD"

operational:
  llm:
    model: "qwen3:8b"
    temperature: 0.5       # low temp = consistent, terse voice
    max_tokens: 1200
  schedule:
    heartbeat_hours: 6
  security:
    enable_preflight: true
    enable_output_safety: true

Then test it immediately:

# Interactive chat — no gateway needed
./chat.sh dusk

# Run LLM personality tests against the spec
pytest tests/ -v -s -m llm -k dusk

# Run as a full agent
python -m overblick run dusk

The framework provides automatically:

  • SafeLLM 6-stage pipeline (preflight, rate limiting, output safety, audit log)
  • LLM routing — Ollama local, Gateway load balancer, or Cloud stub
  • Scheduler with heartbeat and quiet hours
  • Secrets management — Fernet-encrypted, per-identity
  • Web dashboard monitoring at localhost:8080
  • 8-step setup wizard — starts automatically on first run

Setup

Get Started

No npm. No build step. Requires a local or cloud LLM — Ollama, LM Studio, or Deepseek.

1

Clone and install

# Requires Python 3.13+
git clone https://github.com/jensabrahamsson/overblick.git
python3.13 -m venv venv && source venv/bin/activate
pip install -e ".[dev]"
2

Run the test suite

# 3250+ tests — no LLM or browser required
pytest tests/ -v -m "not llm and not e2e"
3

Start everything

# Starts gateway + dashboard and opens your browser
python -m overblick start

# Chat with a personality directly in the terminal
./chat.sh cherry

On first run, the setup wizard configures your LLM, creates an identity, and activates plugins — all from the browser. Supports Ollama, LM Studio, and Deepseek.

◈

After onboarding, verify your identities sound right: pytest tests/ -v -s -m llm

Each identity ships with model-specific voice hints (llm_hints/qwen3_8b.yaml) that reinforce personality traits for that LLM. Running a different model? Create your own hints file — the system auto-loads llm_hints/<model>.yaml based on your configured model.
